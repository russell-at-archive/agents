# CDD Macro: The Manifesto

Contract-Driven Design (CDD) is a strategic approach where the "contract" (the
formal agreement between services) is defined *before* any implementation
begins. This manifesto outlines the core principles of CDD.

## Core Principles

1. **Contract-First over Code-First:**
   Defining the interaction before the implementation ensures that the interface
   is designed for the consumer's needs.
2. **Explicit over Implicit:**
   Every interaction, schema, and error condition must be explicitly defined in
   a machine-readable format.
3. **Independent Evolvability:**
   Services should be able to evolve independently. A strong contract enables
   providers to change internals without breaking consumers.
4. **Robustness Principle (Postel's Law):**
   "Be conservative in what you send, and liberal in what you accept."
5. **Single Source of Truth:**
   The contract is the ultimate authority on how to interact with a service.

## Why CDD?

In modern distributed systems, the primary failure mode is **integration
breakdown**.

- Without CDD: Breakages are only discovered at integration time.
- With CDD: The contract acts as a stable anchor for parallel development.

## Strategic Impact

CDD transforms the SDLC by:

- **Reducing Lead Time:** Teams can develop against mocks immediately.
- **Lowering Rework:** Catching interface mismatches during design is cheaper.
- **Improving Quality:** Automated testing ensures implementation matches spec.
