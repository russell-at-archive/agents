# CDD Meso: The Workflow

Contract-Driven Design (CDD) is a collaborative process that spans design,
development, and testing. This guide outlines the standard CDD workflow.

## The CDD Lifecycle

1. **Collaboration (Discovery):**
   Consumers and providers meet to define requirements using techniques like
   Event Storming or API Design First.
2. **Contract Definition (Design):**
   The contract is authored using machine-readable specifications like
   **OpenAPI (REST)**, **AsyncAPI (Events)**, or **gRPC (Protobuf)**.
3. **Validation & Mocking (Simulation):**
   The contract is validated against style guides (e.g., Spectral) and used to
   generate interactive mocks (e.g., Prism).
4. **Parallel Development (Execution):**
   Consumers build against mocks while providers build the implementation.
5. **Contract Verification (Testing):**
   Automated tests (e.g., Pact, Dredd) ensure the provider satisfies the
   contract and the consumer uses it correctly.

## Consumer-Driven Contracts (CDC) vs. Provider-Driven

While CDD often starts with the Provider-First approach, **CDC** (e.g., Pact)
reverses the flow:

- **Consumer-Driven:** Consumers define exactly what they need.
- **Provider-Driven:** The provider defines the contract, and consumers adapt.

## Versioning & Compatibility

- **Semantic Versioning:** Follow semver.org rules for breaking changes.
- **Backward Compatibility:** Never make breaking changes in-place.
- **Sunsetting:** Communicate and manage the end-of-life for old versions.
