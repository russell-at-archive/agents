# 03 Micro Level: Implementation Patterns

At the micro level, Domain-Driven Design (DDD) provides a set of patterns
for building a rich, expressive domain model that captures complex business
logic in code.

## 1. Aggregates and Aggregate Roots

An **Aggregate** is a group of related objects (Entities and Value Objects)
that are treated as a single unit for data changes.

- **Aggregate Root:** The only member of an aggregate that can be directly
  accessed from outside. All interactions must go through it.
- **Invariants:** Business rules that must be true at all times. The
  Aggregate Root is responsible for enforcing these invariants.
- **Transactional Boundary:** Changes to an aggregate should be saved or
  rolled back as a single atomic operation.

## 2. Entities vs. Value Objects

- **Entities:** Have a unique identity (e.g., `UserId`, `OrderId`). They
  are mutable, but their identity stays the same.
- **Value Objects:** Have no identity. Two value objects are equal if their
  properties are the same (e.g., `Amount(10, "USD")`). They are immutable.

## 3. Repositories

A **Repository** is an abstraction for retrieving and persisting aggregates.

- It provides a collection-like interface (e.g., `FindById`, `Save`).
- The domain layer defines the repository **interface (Port)**.
- The infrastructure layer provides the repository **implementation (Adapter)**
  (e.g., using an ORM or raw SQL).

## 4. Domain Events

A **Domain Event** is a record of something significant that happened in the
business (e.g., `OrderPlaced`, `PaymentReceived`).

- Events are used to decouple different parts of the system.
- An aggregate can publish a domain event after a state change.
- Other parts of the system can listen for these events and react.

## 5. Domain Services vs. Application Services

- **Domain Services:** Contain logic that doesn't belong to any single Entity
  or Value Object (e.g., a "TransferFundsService").
- **Application Services:** Orchestrate use cases. They don't contain
  business logic. They fetch aggregates, call methods, and save.

## 6. Implementation Checklist

- Is the code using the **Ubiquitous Language**?
- Are business rules (invariants) encapsulated within the **Aggregate Root**?
- Are you using **Value Objects** for concepts like money or addresses?
- Is the **Domain Layer** free of infrastructure dependencies?
- Are **Application Services** kept thin, delegating work to the domain model?
