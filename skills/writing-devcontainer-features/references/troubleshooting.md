## Common mistakes

### Missing `set -e`

**Symptom:** Feature appears to install successfully but tool is missing or
broken at runtime.

**Cause:** Without `set -e`, a failed `curl`, `tar`, or `apt-get` step is
silently ignored and execution continues.

**Fix:** First line after the shebang must be `set -e`. No exceptions.

---

### Options defined but not consumed

**Symptom:** Changing an option value has no effect.

**Cause:** Option key in `devcontainer-feature.json` does not match the env
var name used in `install.sh`.

**Rule:** Option key → uppercased env var. Key `nodeVersion` → `NODEVERSION`.
Key `install-extras` → `INSTALLEXTRAS`. The mapping is all-caps, no hyphens
or separators.

**Fix:** Check every option in the JSON against the env var read in the
script. Add a fallback default in the script: `FOO="${FOO:-"default"}"`.

---

### Hardcoded architecture

**Symptom:** Feature fails on arm64 (Apple Silicon, Graviton) with a 404 or
binary execution error.

**Cause:** Download URL has a literal `amd64` or `x86_64` string.

**Fix:** Always detect with `uname -m` and map to the download format:

```bash
ARCH="$(uname -m)"
case "${ARCH}" in
  x86_64)  ARCH="amd64" ;;
  aarch64) ARCH="arm64" ;;
  *) echo "(!) Unsupported: ${ARCH}"; exit 1 ;;
esac
```

---

### No OS detection

**Symptom:** Feature fails with "command not found: apt-get" on Alpine or
"command not found: apk" on Debian.

**Fix:** Source `/etc/os-release` and branch on `$ID`:

```bash
. /etc/os-release
case "${ID}" in
  ubuntu|debian) ... ;;
  alpine) ... ;;
  *) echo "(!) Unsupported OS: ${ID}"; exit 1 ;;
esac
```

---

### Non-idempotent install

**Symptom:** Re-running the feature (e.g. during `dependsOn` resolution or
duplicate test) causes errors like "directory already exists".

**Fix:** Guard installation steps with existence checks:

```bash
if ! command -v mytool &> /dev/null; then
  # install
fi
```

For directories: `mkdir -p` instead of `mkdir`.

---

### `installsAfter` used for a hard dependency

**Symptom:** Feature fails when the dependency is not explicitly included by
the user.

**Cause:** `installsAfter` is a soft ordering hint — it only reorders features
already queued. It does not force installation.

**Fix:** Use `dependsOn` for hard dependencies:

```json
{ "dependsOn": { "ghcr.io/devcontainers/features/node:1": {} } }
```

## install.sh failures

### curl: (22) The requested URL returned error: 404

- Version string is wrong — `latest` may need resolving via GitHub API.
- Architecture not mapped correctly — check the exact filename pattern from
  the upstream release page.
- HTTPS redirect not followed — ensure `-L` flag is included in `curl -fsSL`.

### tar: unexpected end of file

- The download itself failed (e.g. 403, wrong URL) but the exit code was not
  caught. Ensure `set -e` is present and the `curl | tar` pipeline propagates
  failures: add `set -o pipefail` after `set -e`.

### permission denied when running installed binary

- Binary was installed without execute permissions.
- Fix: `chmod +x /usr/local/bin/mytool` or use `install -m 0755`.

## Test failures

### `reportResults` not called

**Symptom:** Test script exits 0 but produces no output, or the test framework
marks it as failed.

**Fix:** Always end every test file with `reportResults`.

### Scenario `.sh` file missing

**Symptom:** `devcontainer features test` warns about a missing test file for
a scenario.

**Fix:** Every key in `scenarios.json` must have a matching `<key>.sh` file
in the same directory.

### Test passes locally but fails in CI

- CI runner may be amd64-only; test arm64 scenarios with `--base-image`
  against a multi-platform image or use a matrix.
- CI may time out on large image pulls; use slim/minimal base images in tests.
- Ensure `source dev-container-features-test-lib` path resolves in CI — it is
  injected by the devcontainer CLI, not present in the repo.

## Publish failures

### 403 when pushing to GHCR

- The GHCR package does not exist yet and the token lacks `packages:write`.
- Ensure `permissions: packages: write` is set in the GitHub Actions job.
- After first publish, set the package to **public** in GHCR package settings.

### Feature not updated after code change

- The `version` field in `devcontainer-feature.json` was not bumped.
- Features are only republished when `version` changes.

### `devcontainer-collection.json` is stale

- This file is auto-generated by `devcontainers/action` — do not commit it
  manually. Delete any local copy and let the Action regenerate it.

## Anti-patterns

| Anti-pattern | Correct approach |
|---|---|
| `chmod +x install.sh` in CI | CLI invokes the script directly; execute bit is set by the packaging tool |
| Installing to `$HOME` during `install.sh` | `install.sh` runs as root; use `_REMOTE_USER_HOME` for user-space files |
| Using `sudo` inside `install.sh` | Script already runs as root; `sudo` is unnecessary and may not be installed |
| Fetching secrets or tokens in `install.sh` | `install.sh` runs at build time; secrets should be injected via `remoteEnv` at runtime |
| Putting large binaries in the feature tarball | Download at install time; keep the feature tarball small |
| Writing to `/root` for the remote user | Use `_REMOTE_USER_HOME`; the remote user may not be root |
| Omitting `--no-install-recommends` on apt | Bloats the image layer; always use `--no-install-recommends` |
